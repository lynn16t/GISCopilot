API 层 · ModelProvider
底层通信
▶

所有 LLM 调用都经过统一接口 unified_llm_call()，内部路由到具体 Provider。

支持的 Provider
OpenAIProvider OllamaProvider （可扩展）

路由依据：
config.ini → API_Key 段决定使用哪个 provider。
模型名 → ModelProviderFactory._model_providers 映射。
统一调用签名
unified_llm_call( request_id, # 本次请求追踪ID messages, # List[{role,content}] model_name, stream, # True/False reasoning_effort # gpt-5系列专属 )
Streaming 处理
  • stream=True：逐 chunk 拼接，通过 AgentWorkerThread.output_written 信号实时推到 GUI
  • stream=False：等待完整回复（code_review 步骤使用）
  • OpenAI 格式：读 chunk.choices[0].delta.content
  • GPT-5 格式：额外兼容 chunk.delta.content / chunk.content / chunk.output
reasoning_effort：仅对 gpt-5 / gpt-5.1 / gpt-5.2 生效，其他模型自动剥离该参数。
LLM 调用重试机制（get_LLM_reply）
  • retry_cnt = 3，失败后 sleep 10s 再重试
  • 仅用于老版直接调用路径；unified_llm_call 不含重试（由上层 execute_complete_program 管理）

对话循环 · OutputParser · GuardGate
分类 & 路由
▶

Phase 5 核心机制：不做意图预分类，所有消息直接喂给 LLM，由 LLM 输出格式决定后续路由。

Prompt 构成（conversation 步骤）
System Message（静态部分 — 可命中 Prompt Cache）
【架构提示词】CONVERSATION_SYSTEM_PROMPT（Constants.py）
定义 5 种响应 Mode：
Mode 1: TASK_SUMMARY（任务摘要+请求确认）
Mode 2: [TASK_CONFIRMED]（确认触发分析）
Mode 3: CLARIFYING_QUESTION（反问缺失信息）
Mode 4: CONVERSATIONAL_REPLY（普通对话）
Mode 5: [KNOWLEDGE_UPDATE]（建议写入知识库）
【架构提示词】General GIS/QGIS Rules（SessionContext._get_general_rules）
· 始终用 processing.run() · from qgis.core import * · 用绝对路径 · 验证图层有效性
【知识注入】Project Knowledge（knowledge_manager.get_relevant_knowledge）
根据当前图层名 + 任务描述检索知识库，max 6000字符
System Message（动态部分 — "---CONTEXT---" 分隔后）
【动态注入】=== Loaded Data ===（当前 QGIS 图层列表，每次对话前本地刷新）
【动态注入】=== Current Plan ===（若存在 current_plan）
【动态注入】=== Recent Execution Results ===（最近3条执行结果）
History（压缩后的对话历史）
· 第一条消息（原始任务）始终保留
· 中间消息：[Plan] 删除 / [Result] 保留 / [Executed code] → "[Code was executed]" / 普通消息截断到200字
· 最近6条完整保留
User Message
用户当前输入的原始消息
OutputParser 分类规则（纯正则/启发式，不依赖 LLM 打标签）
CODE
含 ```python 代码块
PLAN
含 {"steps": [...]} JSON 结构
GIS_TASK_READY
含 [TASK_CONFIRMED] 标签
KNOWLEDGE_UPDATE
含 [KNOWLEDGE_UPDATE] 标签
QUESTION
含问号 / 疑问词 / 反问特征
CHAT
以上都不匹配，默认类型
GuardGate 路由决策
  • CODE → action: show_code → 代码发到 CodeEditor，聊天框显示提示
  • PLAN → action: confirm_plan → 格式化展示 plan，状态机进入 PLAN_READY，等待用户确认
  • GIS_TASK_READY → action: trigger_analysis → 提取 refined_task，触发完整分析管线
  • KNOWLEDGE_UPDATE → action: confirm_knowledge → 请求用户确认后写入知识库
  • QUESTION / CHAT → action: show_message → 直接显示到聊天框
反问确认流程：LLM 输出 Mode 3（CLARIFYING_QUESTION）时，OutputParser 识别为 QUESTION，GuardGate 路由到 show_message + is_question=True，回显到聊天框等待用户补充。用户补充后重新进入对话循环，LLM 收到完整信息后输出 Mode 1（任务摘要），再次等待确认后输出 Mode 2（[TASK_CONFIRMED]）。
→
GIS_TASK_READY 路径传入分析管线
refined_task（[TASK_CONFIRMED] 之后的精炼任务描述字符串）
→ 触发 _run_analysis(refined_task)

步骤 3.5 · Embedding 候选工具检索
语义检索
▶

用 task_breakdown 在本地向量索引中检索语义最相关的工具候选，无 LLM 调用，纯本地计算。

索引构成（启动时一次性构建）
TOML 文件工具（高质量）
来源：Tools_Documentation/QGIS_Tools/*.toml
embed 文本：tool_name + ". " + brief_description（截断到500字）
Registry 补充工具（低质量）
来源：qgis340_tools.json（无 TOML 的工具）
embed 文本：只有 tool_name，无描述
检索引擎（优先级）
主路径：ONNX（语义相似度）
模型：all-MiniLM-L6-v2（384维，本地CPU）
query → 384维向量 → 矩阵点积（cosine，已L2归一化）→ argsort top-20
语义匹配：buffer ≈ zone of influence ≈ surrounding area
备选路径：TF-IDF（关键词匹配）
无 onnxruntime/tokenizers 时自动降级
sparse cosine similarity，只能匹配共现词
关键词匹配：必须词面重叠
TOOL_WHITELIST（固定常用工具，始终带入）
不参与排名，直接合并进候选列表排在前面。覆盖高频 GIS 操作：
reprojectlayer buffer clip intersection dissolve joinattributesbylocation extractbyattribute fieldcalculator zonalstatisticsfb rastercalc … 共约40个
top-20 无分数阈值过滤，低相似度工具也会进入候选。最终由下一步 LLM 工具选择过滤。
→
传入下一环节
candidate_tools_str：whitelist工具 + top-20检索工具，格式化为 "- ToolName (ID: xxx): desc" 列表

步骤 7c · 代码执行 + 自动调试（2轮）+ Post-Debug Review
执行层
▶
执行循环（execute_complete_program，try_cnt=2）
while count < 2: try: compile(code, 'Complete program', 'exec') exec(compiled_code, globals()) # stdout 重定向到 StringIO return code, user_output, [] # 成功 → 直接退出 except Exception as err: # 记录 error_collector: attempt/code_snapshot/error_message/traceback if count == 2: return code, output, error_collector # 最终失败 # AI 调试 → 生成新代码 → 下轮执行
AI 调试 Prompt（每次失败后，非最后一轮）
System Message（新模式：SessionContext）
【架构提示词】debug_role：具备 SmartDebugger 能力的GIS专家
【知识注入】Project Knowledge
【动态注入】=== Current Plan ===
【动态注入】=== Recent Execution Results ===（最近3条）
User Message（build_debug_instruction 构建）
【架构提示词】debug_task_prefix（修正代码并返回完整修正版）
Error message: {str(err)}
Failed code: {完整代码块}
Tool documentation: {documentation_str}
Requirements:（get_smart_debug_requirements 动态生成）
基础要求（固定）：
· 分析错误模式 · 返回完整代码块 · 可用 geopandas/numpy 等
· shapefile字段名≤10字符 · 不用临时图层作输出
Smart Debug 建议（动态，来自 SmartDebugger 历史错误模式匹配）：
· 根据当前错误类型从历史学习库中推断最可能的修复建议
API调用失败：continue（用原代码重跑，不消耗调试次数）
Post-Debug Review（2次全失败 + is_review=True）新增

2轮调试仍失败后，把最终失败代码再过一次 Code Review，然后执行1次（try_cnt=1，不再调试）。

  • Prompt 与步骤 7b 完全相同（build_code_review_instruction）
  • is_review=False 时跳过，直接进入 result_ready
  • final run：try_cnt=1，review=False，无调试
执行成功判定
execution_success = ( code is not None and len(code.strip()) > 0 and (len(error_collector) == 0 or output) )
结果写入 SessionContext
  • session.add_executed_code(final_code) → messages 中记录 [Executed code]
  • session.add_result(result_str) → messages 中记录 [Result]（含 Status/Output/Error）
  • 状态机 → RESULT_READY，emit result_ready 信号
→
最终输出
result_info：{ code, success, output, error_message, error_collector }
→ GUI 展示结果，状态机进入 RESULT_READY 等待用户反馈
